Guia Detalhado — Git & GitHub

Conceitos, fluxos de trabalho e resolução de conflitos explicados em profundidade

Conceitos Fundamentais de Git & GitHub

O Git é um sistema de controle de versão distribuído. Isso significa que cada desenvolvedor tem uma cópia completa do repositório, incluindo o histórico, no seu computador. Com isso, é possível trabalhar offline, criar experimentos em branches e integrar mudanças de forma colaborativa.

Principais Conceitos

  • Commit: uma foto do seu código em determinado momento. É importante escrever mensagens claras, por exemplo: feat: adiciona página inicial.
  • Branch: uma ramificação do projeto. Serve para desenvolver algo novo sem atrapalhar a versão estável.
  • Merge: processo de unir mudanças de duas branches.
  • Rebase: forma de reaplicar commits em cima de outra base, mantendo histórico linear.
  • Remote: repositório hospedado em um servidor (GitHub, GitLab, Bitbucket).

GitHub

O GitHub é uma plataforma que hospeda repositórios Git. Ele adiciona recursos de colaboração como Pull Requests, Code Review, Issues e GitHub Pages (para deploy de sites estáticos).

Fluxos de Trabalho

Um fluxo define como a equipe organiza suas branches e integra mudanças. O objetivo é evitar caos no histórico e garantir qualidade do código.

GitHub Flow (Simples)

  1. Branch principal: main sempre estável.
  2. Para cada tarefa, cria-se uma branch: feat/nova-funcionalidade.
  3. Após o desenvolvimento, abre-se um Pull Request para main.
  4. O código passa por revisão e testes.
  5. Se aprovado, faz-se merge e deploy.
Esse fluxo é usado em projetos menores, onde a integração é contínua.

Git Flow (com Develop)

Usado em times maiores. As branches principais são:

  • main: versão de produção.
  • develop: integração de features.
  • feat/*: desenvolvimento de novas funcionalidades.

No final, develop é mesclada na main para gerar uma versão.

Resolução de Conflitos

Conflitos acontecem quando duas branches modificam a mesma linha de um arquivo. O Git não sabe qual versão manter, então pede que o desenvolvedor escolha.

Exemplo de Conflito

<<<<<<< HEAD
console.log("Olá da sua branch");
=======
console.log("Olá da branch que veio do merge");
>>>>>>> develop

Nesse caso, você precisa editar o arquivo e deixar apenas a linha correta, por exemplo:

console.log("Olá final");

Passo a Passo para Resolver

  1. Rode git status para ver quais arquivos estão em conflito.
  2. Abra os arquivos e procure os marcadores <<<<<<<, ======= e >>>>>>>.
  3. Escolha qual parte do código manter ou mescle as duas versões.
  4. Apague os marcadores e salve o arquivo.
  5. Marque como resolvido: git add arquivo.
  6. Finalize o merge com git commit ou continue o rebase com git rebase --continue.

Dicas Importantes

  • Faça git pull frequentemente para evitar grandes divergências.
  • Mantenha commits pequenos e focados, isso facilita encontrar onde ocorreu o conflito.
  • Use ferramentas visuais de merge (VS Code, GitKraken, Sourcetree) para simplificar a resolução.

Pull Requests

Um Pull Request é uma solicitação para mesclar uma branch em outra. Ele permite revisão do código por colegas, testes automatizados e discussão sobre a mudança.

Checklist de um bom PR

  • Título claro e descritivo: feat(login): adiciona autenticação de usuários.
  • Descrição do que foi feito e por que.
  • Testes realizados localmente.
  • Sem arquivos irrelevantes no commit.

Deploy com GitHub Pages

Após integrar tudo na main, é possível publicar sua página HTML de forma gratuita.

  1. No repositório, vá em Settings > Pages.
  2. Selecione Deploy from a branch.
  3. Escolha a branch main e a pasta / (root).
  4. O GitHub gerará uma URL no formato https://usuario.github.io/repositorio/.
